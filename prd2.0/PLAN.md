# J-Pop 信息中心 — 产品规划 v0.1

## 产品愿景
为中文 J-Pop 受众（包括不会日语的新观众）打造一个**聚合、可追踪、语言友好**的信息中心。首期聚焦“乐队信息”，之后再扩展到大事件、爱豆团、个人主页等板块。

## 第一阶段已确认的需求
1. **语言**：网站 UI 支持中 / 日 / 英三语切换。
2. **信息源**：乐队/艺人 Twitter/X 动态 + 日本娱乐新闻并重。
3. **Twitter 数据方案**：由于 X 官方开发者 API 昂贵且门槛高，首期采用 **Nitter RSS + 新闻 RSS** 作为替代方案。
4. **“想看巡演”**：用户点击“想看”后保存，后续推送相关消息。
5. **首期范围**：先做“乐队信息”板块，而非全行业大事件流。
6. **设计风格**：参考 Apple 官网，使用主色调：
   - Khaki Green: `#6F8436`
   - Crystal Yellow: `#E5D89E`
   - Gold Ore: `#EDBC13`
   - Organic Garden: `#ABAB64`
   - Hinterlands Green: `#38460C`

## 第一个版本（MVP）要做什么

### 1. 整体结构
- **多语言首页**：干净、Apple 风格，顶部导航 + 语言切换器。
- **乐队列表页**：卡片展示精选乐队，点击进入详情。
- **乐队详情页**：一个乐队一个页面，包含：
  - 乐队简介与官方链接
  - 最新动态（来自官方 X / Nitter RSS）
  - 日本音乐新闻聚合（来自 Natalie / Oricon 等 RSS）
  - 巡演信息（首期手动维护，支持“想看”收藏）
  - 最新作品/专辑信息（首期手动维护）
- **“我的想看”**：用户收藏的巡演列表，数据先存在浏览器本地。

### 2. 不做的事（放到后续版本）
- 用户登录/账号系统
- 内容自动翻译（首期内容以日文原文为主，UI 三语切换）
- 推送通知（可以用邮件/推送替代，但需账号系统）
- 大量乐队/爱豆团（先精选 2-3 个乐队验证模式）

## 技术实现思路（大白话）

- **前端框架**：Next.js 14 + React + TypeScript。用 Next.js 是因为它自带“后端小接口”能力，可以安全地从服务器去抓 RSS/Nitter，避免浏览器直接访问被屏蔽。同时部署方便。
- **样式**：Tailwind CSS + shadcn/ui。Tailwind 写响应式快，shadcn 提供卡片、按钮、标签等现成且好看的组件。
- **字体**：使用 Inter（免费，最接近 Apple 的 San Francisco）。
- **多语言**：用 `next-intl` 或自建的简单语言上下文，把页面文案翻译成中/日/英。
- **数据源**：
  - **新闻**：用 `rss-parser` 在 Next.js API 路由里读取 Natalie.mu / Oricon 的 RSS。
  - **X 动态**：通过 Nitter 的 `https://nitter.net/{用户名}/rss` 抓取。Nitter 是第三方镜像，不稳定，所以代码里会加“失败时优雅降级”逻辑。
  - **巡演/专辑**：先做成可配置的数据文件（JSON），方便你随时增删改。
- **收藏功能**：用浏览器 `localStorage` 存“想看的巡演”，页面刷新后还在。后续加账号登录后再迁移到数据库。
- **部署**：Vercel（免费），可以直接绑定自定义域名。

## 复杂度评估

**中等**。

- 不算简单：需要处理多语言、RSS 解析、错误降级、响应式 UI。
- 但也不是大工程：没有复杂后端、没有数据库、没有账号系统，适合快速做出可发布的产品。

预计构建时间：一轮密集开发即可完成 MVP。

## 关键风险（提前说清楚）

1. **Twitter/X 数据不稳定**：Nitter 是民间镜像，可能失效或变慢。如果彻底不可用，官方 X 动态会退化为“手动维护的最新消息”或改用 Bluesky/Instagram 等公开源。
2. **RSS 源变化**：新闻站 RSS 结构可能变化，需要偶尔维护。
3. **内容翻译**：自动翻译新闻/推文需要额外付费 API（如 DeepL）。本期只翻译 UI，内容保留原文，避免产生成本。
4. **版权问题**：只展示标题、摘要、链接，跳转原文。不存储全文，不重新发布。
